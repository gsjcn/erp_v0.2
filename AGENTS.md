## 1. 阶段边界

当前仍是第一阶段。只有用户明确说出：

```text
第一阶段 完成，可以开始下一阶段
```

才允许进入下一阶段。

第一阶段允许范围：

- 客户
- 零件管理 / 产品零件库
- 订单 / 下单
- 组件 / 零件清单 Excel 导入
- 零件基础库、零件搜索、适用范围、机型零件包 / BOM、图纸版本、来源加工关系
- 每个订单零件独立选择生产流程
- 流程记忆 / 标准工序维护
- 生产
- 仓库
- 库存
- 只读统计

第一阶段严禁范围：

- 质量、检验、追溯、IATF 16949、APQP、PPAP、FMEA、MSA、SPC、8D、CAPA
- 财务、采购、供应商、审批流、复杂权限、成本、开票
- 报表中心、驾驶舱、复杂排产、设备负荷、工时绩效
- 自动提交生产、自动扣库存、自动创建生产任务、自动把建议保存为正式规则

第一阶段主流程：

```text
维护客户 -> 维护零件基础资料 -> 客户下单 -> 选择或导入订单零件
-> 每个零件选择生产流程 -> 人工提交生产 -> 生产确认
-> 仓库入库 / 发货 -> 库存查看 -> 只读统计
```

零件管理是客户和订单之间的基础资料控制面板，不是订单新增弹窗的附属功能。订单新增弹窗只调用零件管理维护好的零件、零件包和默认图纸，不承担大量基础资料维护。

## 2. 当前开发顺序

开发必须按下面顺序推进，不得跳过前置数据基础直接做自动化建议。

1. 零件基础库单个维护：查询、新增、编辑、软停用、订单可直接调用。
2. 零件适用范围：全部客户、指定客户、全部机型 / 项目、指定机型 / 项目、指定客户 + 指定机型 / 项目。
3. 零件管理控制面板：放在客户和订单之间，集中查看全部客户、全部机型 / 项目、全部零件、最近下单日期、图纸状态和默认工艺。
4. 机型零件包 / BOM：维护 B3、B5 等标准清单，区分百胜通用款、客户通用件和客户机型定制件。
5. 零件库 Excel 导入：独立模板、上传预览、校验、人工确认、事务写入；可从订单 `ERP上传净表` 提取零件包草稿，但不能直接覆盖正式 BOM。
6. 订单页面接入推荐：按客户 + 机型 / 项目推荐零件包，允许一键带入 `DRAFT` 订单明细。
7. 来源加工关系：维护百胜通用件 / 半成品加工成客户零件的关系，只做建议，不自动扣库存。
8. 自动建议：基于历史订单推荐常用零件、机型零件包或来源加工关系，但必须人工确认后才保存为正式规则。

P0-P5 任务计划：

- P0：稳定性和数据安全。项目必须能启动、构建、连接 PostgreSQL；migration 可重复部署；任何零件、BOM、图纸和导入功能不得自动提交生产、自动扣库存或物理删除历史数据。
- P1：零件管理控制面板。新增左侧菜单入口，位于客户和订单之间；支持全部零件查看、客户筛选、机型 / 项目筛选、通用件 / 定制件筛选、图号 / 图纸日期 / 图纸状态筛选、最近下单日期展示、零件新增 / 编辑 / 软停用。
- P2：零件包 / BOM。支持百胜通用 BOM、客户通用 BOM、客户指定机型 BOM；支持从百胜通用款复制为客户 BOM，复制后独立维护；BOM 行能维护默认数量、零件类型、默认图纸版本和默认工艺。
- P3：订单和 Excel 接入。订单新增只调用零件管理成果；按客户 + 机型推荐 BOM 并带入 `DRAFT` 明细；订单 `ERP上传净表` 可提取为零件 / BOM 草稿，必须预览和人工确认，不得直接覆盖正式资料。
- P4：图纸版本和工序关系。零件必须支持多图纸版本；BOM 行可指定默认生产图纸；默认下单使用当前启用最新图纸或 BOM 指定图纸；工序默认值可来自零件或 BOM 行，但下单后仍允许人工调整并留痕。
- P5：体验优化和自动建议。桌面端优先做筛选、批量维护、复制 BOM、差异提示和导出；手机端订单保持只读，生产和仓库现场执行允许手机操作。自动建议排最后，只能生成待确认建议，不得自动保存正式规则。

## 3. 技术和部署

推荐技术栈：

- `database`: PostgreSQL 16
- `ORM`: Prisma
- `backend`: NestJS + TypeScript + Node.js
- `frontend`: Vue 3 + TypeScript + Vite + Element Plus
- `deployment`: Docker Compose
- `storage`: NAS 本地目录，后续可迁移到 MinIO / S3

原则：

- 正式数据库必须使用 PostgreSQL，不得用 JSON 文件、SQLite 或 MongoDB 作为正式主库。
- 前后端必须使用 TypeScript。
- 数据库访问统一通过 Prisma。
- 库存变化必须通过 `InventoryTransaction` 追加流水，不得静默改总数。
- 业务表保留稳定编号，例如 `orderNo`、`partCode`、`productionTaskNo`。
- `quantity` 和 `unit` 必须分字段。
- 业务文本字段不得随意增加 `MaxLength` 或 `@db.VarChar`，避免 Excel 长备注、工艺路线、产品名称、规格被截断。

NAS 部署：

- PostgreSQL、backend、frontend 使用 Docker Compose 分容器部署。
- PostgreSQL 不直接暴露公网，只允许 backend 访问。
- 上传、导出、备份目录通过 `.env` 配置，不得写死。
- 仓库必须维护 `.dockerignore`，避免提交或打包 `node_modules`、日志、数据库目录、上传文件和导出文件。

## 4. UI 和跨端

- 桌面端是第一阶段主要业务操作端。
- 手机端订单只保留订单列表 / 订单详情查看，不开发订单新增、编辑、Excel 导入、上传、提交生产。
- 手机端生产允许现场执行：车间主任可开始生产、批量开始、确认完成、处理生产通知和补单审核；车间工人可填写当前工序完成记录。
- 手机端仓库允许现场执行：仓管可确认入库、按单批 / 整单 / 批量确认发货、处理仓库通知；新增仓库、库位和基础配置仍在电脑端操作。
- 手机端暂不开发扫码功能，但生产和仓库页面的操作结构要预留后续扫码入口。
- 所有页面的通知入口必须能查看历史消息；通知必须按业务归类展示，至少区分生产通知、仓库通知和管理员全部通知。
- 管理员可查看全部历史消息；生产角色只查看生产相关消息，例如新订单、生产变更、生产完成、补单；仓库角色只查看仓库相关消息，例如生产完成待入库、发货、库存处理、取消/变更转库存。
- 通知确认只把状态从 `PENDING` 改为 `ACKNOWLEDGED`，不得删除通知；历史消息默认按时间倒序展示，并支持按订单、客户、零件、通知类型和时间筛选。
- 第一阶段不引入复杂权限系统；管理员、生产、仓库的通知可见性先通过通知 `target` 和页面入口过滤实现，不能为了通知中心提前开发完整权限模块。
- 第一阶段暂不开发检验人员功能，但通知和状态流转设计要允许后续在“生产完成 -> 仓库入库”之间插入检验确认节点，避免未来重写主流程。
- Web、Figma 和文档入口保持一致：客户、零件管理、订单、生产流程、流程记忆、生产、统计、仓库、库存。
- 本地前端端口固定 `5176`，Vite 绑定 `0.0.0.0` 并启用 `strictPort`。
- 业务表格必须支持横向滚动，避免文字重叠。
- 关键操作必须用 `el-dialog`，不得用 `window.prompt` / `window.confirm`。
- 搜索、下拉、自动补全不得静默 `take`、`limit`、`pageSize` 或 `slice(0, n)` 截断；需要分页时 UI 必须明确显示总数或继续加载。

## 5. 客户

客户字段：

```text
customerCode, customerName, contactName, contactPhone, address,
regionType, country, province, state, district, city, detailAddress,
remark, status
```

规则：

- 客户名称和客户ID必须大小写不敏感查重。
- 客户下拉只显示客户名称；搜索可命中客户名称、客户ID、联系人、电话、地区、拼音和首字母。
- 客户搜索排序按命中强度：ID 精确、名称精确、前缀、包含、拼音 / 首字母、联系人命中。
- `CustomerSelect` 不得启用 `default-first-option`；客户搜索结果必须按命中强度排序，避免相似客户回车误选。
- 客户只能软停用，不得删除历史订单、库存、生产记录。

## 6. 订单和 Excel 导入

订单字段：

```text
orderNo, customerId, orderDate, deliveryDate, status, remark
```

订单零件字段：

```text
partCode, partName, partType, rowType, componentCode, parentComponentCode,
drawingNo, drawingVersion, drawingDate, drawingStatus, drawingFileName,
drawingFileUrl, partThickness, partSpecification, projectModel,
quantity, productionPlanQuantity, unit, deliveryDate, processRoute, processRemark
```

订单规则：

- 订单必须支持多个零件，至少保留 1 行。
- 订单号创建后永久占用，取消、归档、删除草稿后也不得复用，除非是未提交生产草稿删除时明确释放订单号占用。
- DRAFT 新增和编辑允许手工修改订单号，保存时后端必须再次查重。
- 从 Excel 导入生成的订单只能是 `DRAFT` 草稿，不得自动提交生产。
- 导入错误的 `DRAFT` 草稿订单必须可编辑、可删除；已提交生产、有生产任务、有库存批次或有库存流水的订单不得直接删除。
- 新增 / 编辑订单里的零件类型只能从固定类型中选择，不得启用 `default-first-option`；成品规格可以手工输入，也不得启用 `default-first-option`。
- `quantity` 表示客户订单数量，`productionPlanQuantity` 表示实际生产计划数量，二者必须分开统计。
- `deliveryDate` 可在订单总表或订单零件填写，页面优先显示零件交期。
- `partThickness` 必填，`partSpecification` 表示成品规格。

组件规则：

- `componentCode` 只在同一订单内有效，不是全局编号。
- 新订单块必须从 `C001` 重新计算，支持到 `C9999`。
- 组件行必须有唯一 `componentCode`。
- 子零件必须通过 `parentComponentCode` 指向当前订单内已存在组件。
- 所属组件下拉只能选择当前订单已有组件编号，不能创建不存在的父组件。
- 零件行不能填写自己的 `componentCode`。
- 组件编号被修改时，只能同步仍指向旧组件的子零件；已经手工指向其他组件的子零件不得覆盖。
- 组件行编号被修改时，仍指向旧组件编号的子零件必须自动同步到新组件编号。
- 组件行切换为零件行时，仍指向该组件的子零件必须清空所属组件。
- 如果组件行被切换为零件行，仍指向原组件编号的子零件必须自动清空所属组件。

Excel 导入规则：

- 订单导入只读取 `ERP上传净表`，不得回退读取第一张表。
- 台账表只用于人员录入和复核，不能直接上传。
- 导入入口以 Excel 文件上传为主，支持拖拽上传和多文件连续上传。
- 同一个导入会话可一次多选或分多次上传文件，系统按会话合并校验。
- 上传后必须预览、展示错误 / 警告、支持勾选可导入订单、人工确认提交。
- 提交必须携带 `previewToken` 或等价版本令牌；文件列表变化后必须刷新预览。
- 单文件解析和导入行写入必须事务化；失败要删除临时文件。
- 未提交会话支持删除单个文件和放弃本次导入。
- 已提交会话支持删除导入记录 / 上传记忆，但不得删除已经生成的订单。
- 后端必须重新解析和校验数量，不能只相信 Excel 公式。
- 后端兜底绑定所属组件时，只允许在“上方最近组件 + 当前零件有单套用量 + 当前零件无订单数”的场景绑定。
- 上传文件名、来源文件名、Excel 中文单元格内容必须按 UTF-8 / `exceljs` 正常处理，不得出现中文乱码。
- 订单详情必须能查看来源 Excel 预览；删除导入记忆后，订单保留来源文字，但提示原文件不可预览。
- 导入问题明细必须可导出 Excel，包含订单编号、来源文件、来源行、自动序号、物料号、组件关系、严重程度和问题说明。

## 7. 零件基础库和物料搜索

`Material` 规则：

- `Material` 只保存零件身份基础字段，不保存库存数量。
- `Material` 属于下单搜索记忆，必须支持编辑和软停用。
- 删除只能软停用 `Material`，不得删除历史订单、库存批次、生产任务或导入追溯。
- 订单导入或订单录入新零件时，可以 `upsert` 到 `Material`，但这只代表基础资料入库，不代表库存增加。

搜索规则：

- 零件编码必须支持搜索选择，也必须允许手工输入新编码。
- 搜索大小写不敏感。
- 搜索支持 `partCode`、`partName`、规格、图号、厚度、图纸日期、拼音、首字母、历史客户名称 / 客户名称片段 / 客户拼音首字母 / `customerCode` / 历史订单。
- 新增 / 编辑订单时，零件搜索必须接收当前 `customerId`，优先展示该客户历史用料，并显示最近订单、历史客户和当前客户使用次数。
- 排序必须结合命中强度、当前客户使用次数、当前客户最近订单日期、全局历史使用次数和可用库存。
- `lastCustomerOrderNo` / `lastCustomerOrderDate` 必须优先代表当前客户的最近使用记录。
- 同一 `partCode` 有多个客户历史时，基础名称、规格、图号、版本、厚度和项目型号必须优先采用当前客户最近一次使用记录。
- 新增 / 编辑订单、库存搜索和库存来源核对里的物料建议项必须显示备货库存、订单库存、单位、规格、历史使用次数和命中原因。
- 有当前客户时，空关键字可优先返回该客户历史用料；没有当前客户时不得一次返回全库物料，不得一次返回全部物料。
- 输入关键词后返回全部命中结果，不得静默截断。
- 物料建议项可以摘要展示历史客户列表。
- 物料建议项有图纸日期时必须展示图纸日期。
- 物料快速搜索支持厚度和日期关键词，以及 `2026-05-30`、`2026/5/30`、`2026530` 这类图纸日期关键词查找历史零件。
- 同一 `partCode` 在历史订单中出现多个名称、规格、图号、版本、厚度或项目型号组合时，必须提示同编码多套历史资料风险，让操作员核对图号、规格、厚度和项目型号后再选择。
- 物料建议的冲突风险必须返回并展示具体冲突字段。
- 同一 `partCode` 存在多套历史识别资料时，不得在失焦时自动套用物料资料。
- 操作员选中风险物料后，当前订单行必须持续显示核对提醒。
- 订单详情页必须根据历史订单行重新计算并显示同编码多套历史资料风险。
- 提交生产弹窗必须汇总同编码多套历史资料零件；后端提交生产接口也必须在事务内重新计算同编码多套历史资料风险。
- 操作员确认风险并提交后，摘要本身必须包含提交人员和前几条风险行。
- 操作员未手动选择候选项时，只能在唯一精确命中时自动带出物料资料。
- 如果多个物料同名或多个结果精确命中，必须保留输入并要求操作员选择；多个结果精确命中时，自动带出必须优先按当前输入字段判断唯一命中。
- 自动带出物料资料前必须确认当前输入值仍等于发起查询时的值。
- 选中物料后自动带出资料时，只能在订单行尚未填写图纸资料的字段上补全。
- 选中物料后若手工修改 `partCode` 或 `partName`，必须清理本次自动带出的旧资料。
- 选中历史物料后若操作员又手工修改 `partCode` 或 `partName`，前端必须清理旧物料资料。
- 库存来源核对弹窗里的可替代物料搜索也必须接收当前 `customerId`。
- `sourceType=STOCK` 或 `sourceType=ORDER` 的物料建议只能返回当前范围内有可用数量的物料。
- 库存页的 `Material` 物料基础库搜索也必须支持历史客户、历史订单、图号和项目型号命中。
- 库存来源核对弹窗的替代物料搜索也必须遵守唯一精确命中规则。
- 库存来源替代物料搜索中同一 `partCode` 存在多套历史识别资料时，不得自动切换库存来源。
- 库存来源核对弹窗中，即使接口只返回 1 个模糊命中物料，也不能自动切换库存来源。

零件基础库开发顺序：

- 先做单个维护页面，再做适用范围、零件管理控制面板、机型零件包 / BOM、Excel 导入、订单推荐、来源加工关系、自动建议。
- 零件库导入或单独新增只能写入基础资料和适用规则，不得生成订单、提交生产、占用库存、创建生产任务或改动库存数量。
- 客户适用范围、机型 / 项目适用范围、机型零件包和来源加工关系必须拆成独立关系数据，不得塞进备注字段。

零件管理控制面板：

- 零件管理必须作为独立页面放在客户和订单之间，用于下单前维护基础资料和 BOM，不在订单新增弹窗里承担大量维护。
- 页面上方显示客户筛选和客户名称；选择客户后显示该客户常用项目 / 机型，例如 B3、B5、C型15P。
- 点击项目 / 机型后，下方显示该项目对应的 BOM 零件清单。
- 必须支持筛选：客户、机型 / 项目、零件编码、零件名称、图号、图纸日期、图纸状态、通用件 / 定制件、最近下单日期、是否停用。
- 表格至少显示：零件编码、零件名称、零件类型、通用 / 定制标记、所属客户、所属机型 / 项目、图号、图纸日期、图纸状态、厚度、规格、单位、默认数量、默认工艺、上一次使用下单日期、最近订单号。
- 零件新增、编辑、停用必须复用 `Material` 基础资料和适用范围；停用只影响后续推荐，不得影响历史订单。
- 上一次使用下单日期必须从历史订单行计算，不得手工维护第二份日期。
- 控制面板可以快速查看全部客户的产品，但空筛选不得一次加载无限数据；需要分页或明确的继续加载。
- 任何批量修改都必须先预览差异并人工确认，不得静默覆盖图纸、BOM 或适用范围。

适用范围：

- 支持全部客户通用、指定客户、全部机型 / 项目通用、指定机型 / 项目、指定客户 + 指定机型 / 项目。
- 同一个机型在不同客户下可拥有不同通用件和特定件。
- 客户自己的通用零件可以和百胜公司通用零件不同，不能用全局通用规则覆盖客户特定规则。

机型零件包：

- 机型零件包或等价 `ModelBom` 用于表达 B3、B5 等标准清单，本质按 BOM 管理。
- BOM 类型至少区分：百胜通用 BOM、客户通用 BOM、客户指定机型 / 项目 BOM。
- 通用零件可同时属于多个机型。
- 特定零件只在对应客户 / 机型 / 项目范围内推荐。
- BOM 行必须记录零件、默认数量、单位、零件类型、排序、默认工艺、默认图纸版本、启用状态和备注。
- 可以从百胜通用 BOM 复制生成客户 BOM；复制后客户 BOM 独立维护，不得反向修改百胜通用 BOM。
- 复制关系需要保留来源 BOM，后续百胜通用 BOM 更新时只能提示差异，不能自动覆盖客户 BOM。
- 客户 A 的通用 BOM、客户 A 的 B5 BOM、客户 B 的 B5 BOM 可以同时存在，互不覆盖。
- BOM 行删除只能软停用，不得删除历史订单引用。
- 订单选择客户 + 机型 / 项目后，可以推荐适用零件包并一键带入 `DRAFT` 明细，但不得自动提交生产。

图纸版本：

- 零件必须支持多图纸版本或等价 `MaterialDrawingRevision` 数据，不得只在 `Material` 上覆盖保存一个图号 / 日期。
- 图纸版本至少记录：零件、图号、图纸日期、图纸状态、版本、图纸文件、是否默认、启用状态、备注。
- 默认下单图纸优先级：BOM 行指定默认图纸 > 零件当前启用默认图纸 > 零件当前启用最新图纸。
- 图纸更新必须保留历史版本；旧订单继续指向下单时的图纸快照，不得被新图纸覆盖。
- 修改默认图纸必须记录操作时间和操作人；后续如增加权限，默认图纸修改应限制在电脑端操作。
- 同一零件编码存在多套图纸、厚度、规格或项目型号时，订单新增和提交生产都必须提示风险。

默认工艺：

- 零件或 BOM 行可以维护默认工艺路线，用于下单时带出建议。
- 默认工艺只能作为订单零件流程的初始值，订单保存后每个订单零件仍必须独立拥有自己的工序快照。
- 默认工艺不得绕过生产流程选择页的校验；标准工序名称仍必须来自标准工序库。

来源加工关系：

- 百胜通用件 / 半成品加工成客户零件的关系必须作为 `MaterialTransformRule` 或等价数据维护。
- 至少记录来源零件、目标零件、客户、机型 / 项目、转换说明、默认工艺、损耗或倍率、启用状态。
- 来源加工关系只提供下单和库存来源核对建议，不得自动扣减来源库存。
- 是否使用库存、库存再加工或重新生产，仍由订单提交生产时的库存来源核对决定。

零件库 Excel：

- 零件库批量导入必须使用独立模板，不能复用订单导入净表。
- 建议工作表：`零件基础库`、`适用范围`、`来源加工关系`。
- 后续可增加 `BOM`、`BOM明细`、`图纸版本` 工作表，但必须独立预览和人工确认。
- 必须上传预览、错误 / 警告明细、人工确认提交、事务化写入、失败清理临时文件。
- 必须支持大文件和连续上传，不得限制业务文本长度。
- 同一 `partCode` 出现图号、版本、图纸日期、图纸状态、厚度、规格、项目型号冲突时，必须进入冲突确认流程，不能静默覆盖旧资料。
- 订单 `ERP上传净表` 可以作为“提取零件 / BOM 草稿”的来源，但不能直接当作零件管理正式导入模板。
- 从 `ERP上传净表` 提取出的零件、图纸和 BOM 必须先进入草稿预览，展示与现有零件库、图纸版本和 BOM 的差异，再由人工确认写入。
- 从订单净表提取 BOM 时，组件结构只能形成 BOM 草稿，不得自动创建订单、提交生产、占用库存或改动正式 BOM。

## 8. 生产流程和流程记忆

- 每个订单零件必须独立选择生产流程，不同零件可不同。
- 一个订单零件可选择多个工序，工序可排序。
- 同一个订单零件内不允许重复保存相同工序名；业务上需要多次同类工序时，用 `processRemark` 记录次数或参数。
- 标准工序名称只能选择标准工序，不允许随意手输新名称。
- 生产流程选择页、流程记忆编辑页和订单补单物料流程编辑页的工序排序必须支持拖拽调整；上移 / 下移可以保留为备用操作，但不得作为唯一排序方式。
- 拖拽排序手柄必须使用图标按钮，避免整行拖拽影响选择框和输入框操作。
- 拖拽排序结束后必须清理工序下拉筛选关键字，避免排序后残留过滤条件导致误选。
- 流程记忆只保存 `templateName`、steps、remark 等可复用组合，不绑定客户、订单或零件号。
- 订单页面、生产流程页面和流程记忆页面不得维护各自独立的工序列表。
- 第一阶段生产流程填写人员由下单 / 计划管理人员填写，技术工艺人员作为后续阶段预留。

## 9. 提交生产、生产和变更

履约方式：

- `PRODUCTION`: 重新生产，按 `productionPlanQuantity` 生成 `ProductionTask`。
- `STOCK`: 优先使用备货库存；库存不足部分进入生产计划。
- `REWORK`: 使用已有库存或订单库存再加工，必须选择来源库存。

提交生产：

- Excel 导入、零件库推荐、机型零件包推荐都只能形成 `DRAFT`，提交生产必须由 ERP 订单页面人工触发。
- 提交生产前，后端必须基于数据库最新草稿明细重新校验订单行、组件结构、库存来源、选用数量、人工确认和生产计划偏差。
- 同编码多套历史资料风险必须要求操作员确认后才能提交。
- `STOCK` 已选库存覆盖全部客户订单数量时，不生成 `ProductionTask`。
- `STOCK` 只覆盖部分数量时，先扣减库存并转入订单待发货库存，再按剩余数量生成生产任务。
- `productionPlanQuantity` 是建议生产数量默认值，不是上限；少做或多做必须记录操作人员和原因。

生产：

- 生产状态按 `PENDING -> IN_PROGRESS -> WAITING_CONFIRMATION -> COMPLETED -> STORED`。
- 工序完成表填写完成数量、操作人员和备注。
- 后一道工序默认继承上一道有效完成数量。
- 最终完成数量可小于计划数量，但必须填写报废数量并选择补单申请或管理确认缺货完成。
- 最终完成数量可大于计划数量，超出客户订单数量的部分由仓库入库时转备货库存。
- 已经仓库入库的任务不得再修改工序完成表或最终完成数量。

订单变更：

- 已开始生产的订单不能当作草稿修改。
- 补单、客户数量变更、新增物料、取消订单必须走订单变更流程并通知生产 / 仓库。
- 订单取消、客户数量变更、生产报废补单、管理撤回和通知确认必须保留记录。

## 10. 仓库和库存

仓库规则：

- 仓库编码、库位编码大小写不敏感查重。
- 仓库和库位必须支持新增、编辑、启用、停用。
- 仓库和库位删除必须区分空记录和历史记录：没有 `InventoryBatch`、`InventoryTransaction`、订单或生产引用的空仓库 / 空库位可以删除；已有历史记录的仓库 / 库位只能停用，不得物理删除。
- 仓库和库位的删除、停用、启用必须用 `el-dialog` 二次确认，提示是否会影响后续入库 / 发货选择；不得用 `window.confirm`。
- 仓库停用后，该仓库及其库位不得作为新入库、新转库存、新发货调整的默认候选；历史库存、流水和订单来源仍必须能显示原仓库 / 库位名称。
- 库位停用后不得作为新入库或转库存候选；如果库位下仍有可用库存，库存溯源仍必须展示并允许按业务规则盘点或发货处理。
- 仓库入库必须从已完成生产任务进入，不能手工凭空增加正式库存。
- 入库时必须选择入库数量去向：客户订单数量以内进入订单库存，超出客户订单数量转备货库存。
- 确认发货必须弹窗确认，展示库存批次、订单号、零件、数量、仓库 / 库位，并可填写 remark。
- 发货只减少库存，不修改历史生产完成数量。

库存规则：

- 库存汇总数量必须从 `InventoryBatch` 实时计算，不保存第二份手工汇总。
- 可用库存必须满足 `status=AVAILABLE` 且 `quantity > 0`。
- 盘点改为 0 时批次转 `USED`；0 数量批次盘点为正数可恢复 `AVAILABLE`。
- 盘点必须保留清点人、日期、签字、盘点工单 / 图片 / PDF 附件和库存流水。
- 库存页面必须查看每个零件库存来源批次、生产订单、生产任务、生产日期、客户、仓库、库位、图号、版本和图纸文件。
- 同一零件由多个订单、任务或备货批次叠加时，必须逐批显示，不得只合并总数。
- 库存报警设置属于 `Material` 基础资料，只保存是否启用报警和最小库存数量；不保存库存汇总数量。
- 某些零件允许设置“可有可无”，即不启用库存报警；启用报警的零件必须填写最小库存数量。
- 库存报警只用于提醒和筛选，不得自动下单、自动提交生产、自动补单或自动扣库存。
- 仓库处理不再使用或销毁的库存时，必须通过盘点调整、报废 / 销毁记录或等价库存处理流程把批次数量降为 0，并追加 `InventoryTransaction`；不得静默删除已有库存流水。
- 库存批次、库存流水、订单库存和生产来源一旦存在历史引用，不得物理删除；如需从日常列表隐藏，应使用状态、归档或停用方式。
- 只有没有订单、生产、库存批次、库存流水或预占引用的无效空记录，才允许物理删除；删除前必须弹窗说明影响范围。
- “删除库存记录”在第一阶段应优先解释为“从日常可用库存列表移除”：通过盘点清零、报废 / 销毁状态或停用基础资料实现；不得删除 `InventoryTransaction`。
- 销毁 / 报废库存必须记录操作人、数量、原因、时间，后续如增加附件能力，应支持现场照片、工单或 PDF。
- `Material` 停用只影响后续搜索和推荐，不等于删除库存；如果零件再也不用，应同时把相关可用库存批次按流程清零或报废。

库存来源核对：

- 库存来源弹窗支持按 `Material` / `InventoryBatch` 关键字、拼音、首字母搜索。
- 操作人员必须可逐批选择库存批次和数量。
- 默认库存推荐顺序为匹配优先，同匹配等级内按当前库存数量从少到多。
- 操作人员可以拖拽调整已选库存使用顺序；上移 / 下移只能作为备用操作。
- 替代库存 `partCode` 与订单零件不同，即使图纸资料看似一致，也必须人工确认。
- `selectedStockSources` 合计数量不得超过本次订单零件需要数量。

## 11. 统计和状态枚举

统计只做只读：

- 年度、季度、月度订单数量
- 生产数量
- 入库数量
- 发货数量
- 当前库存数量
- 订单取消、报废补单、客户数量变更等业务数量

统计口径：

- 客户订单数量和生产计划数量分开统计。
- 订单发货数量和实际生产数量分开统计。
- 当前库存按 `InventoryBatch.quantity` 计算。
- 历史入库和转库存按 `InventoryTransaction IN` 计算。
- 历史发货按 `InventoryTransaction OUT` 计算。

状态枚举：

```text
CustomerOrder.status = DRAFT, PENDING_PRODUCTION, IN_PRODUCTION, COMPLETED, CANCELLED
ProductionTask.status = PENDING, IN_PROGRESS, WAITING_CONFIRMATION, COMPLETED, STORED, CANCELLED
InventoryBatch.status = AVAILABLE, RESERVED, USED, SCRAPPED
ProductionNotice.status = PENDING, ACKNOWLEDGED
ProductionNotice.target = PRODUCTION, WAREHOUSE
ProductionScrapRecord.status = PENDING, APPROVED, REJECTED
ProductionShortageHandling = REPLENISHMENT_REQUEST, REPLENISHMENT, MANAGER_APPROVED
```

不要增加复杂状态，除非用户明确要求。

## 12. 逻辑漏洞修正规则

后续开发前必须优先检查这些风险点：

1. 零件基础资料和库存数量不能混在一起；`Material` 不能保存库存数量。
2. 订单导入、零件库导入、零件包推荐都不能自动提交生产。
3. 来源加工关系不能自动扣库存，只能作为库存来源建议。
4. 组件编号不能全局递增，必须按订单块从 `C001` 重算。
5. 子零件需求数量必须按所属组件实际需求计算，不能只按订单总数量或 Excel 公式盲信。
6. 客户特定通用件不能被全局通用件覆盖。
7. 同编码不同图号 / 厚度 / 版本 / 项目型号必须提示冲突，不得静默覆盖。
8. 搜索结果不能隐藏大量候选，不能因为 `limit` 截断导致误选。
9. 中文文件名、来源文件、Excel 中文内容、导入追溯不得乱码。
10. 草稿订单可编辑删除；已产生生产、库存或流水后不得按草稿方式删除。
11. 手机端不得继续扩展订单编辑和上传能力，只保留查看和提示到电脑端操作。
12. 自动建议必须排在最后，只能人工确认后保存为正式规则。
13. 零件管理里的 BOM 复制必须生成客户独立副本，不能让客户 BOM 和百胜通用 BOM 共用同一组可变明细。
14. 默认图纸修改不能覆盖历史订单图纸快照；旧订单必须保留下单时的图号、版本、日期和文件。
15. 从订单 `ERP上传净表` 提取零件包只能生成草稿，不能直接覆盖正式零件库、图纸版本或 BOM。
16. BOM 行默认工艺只能作为下单建议，不能替代订单零件自己的流程快照。
17. 通知确认不得删除消息；生产、仓库和管理员历史消息必须能按归类入口查询。
18. 仓库 / 库位已有库存批次或流水时不得物理删除，只能停用；停用不能影响历史记录显示。
19. 库存报警只能提醒，不得触发自动订单、自动补单、自动生产或自动扣库存。
20. 销毁或不再使用的库存不得直接删除流水；必须通过盘点清零、报废 / 销毁记录或等价流程留痕。
21. 第一阶段不开发检验人员功能，但生产完成到仓库入库之间的数据结构和通知流转必须预留未来检验节点。

## 13. 验收和测试

第一阶段验收标准：

1. 能新增、编辑、停用客户。
2. 能维护零件基础库并在订单中搜索调用。
3. 能在零件管理控制面板按客户、机型 / 项目、通用件 / 定制件、图号、图纸日期和最近下单日期筛选零件。
4. 能维护客户 / 机型 BOM，并能从百胜通用 BOM 复制为客户 BOM。
5. 能维护零件图纸版本，并让 BOM 行指定默认生产图纸。
6. 能按客户 / 机型推荐零件包，但不自动提交生产。
7. 能上传 `ERP上传净表` 创建 `DRAFT` 订单。
8. 能从订单 `ERP上传净表` 提取零件 / BOM 草稿，预览确认后才写入基础库。
9. 能编辑和删除未提交生产的错误草稿订单。
10. 一个订单能录入多个零件，并保留组件 / 子零件结构。
11. 每个零件能选择自己的生产流程。
12. 生产页面能开始生产、逐道工序确认、最终确认完成。
13. 仓库能确认入库和发货。
14. 库存界面能按零件、仓库、客户 / 订单搜索，并显示 0 库存物料。
15. 库存来源核对能逐批选择库存并保留人工确认记录。
16. 仓库 / 库位能编辑、启用、停用；空仓库 / 空库位可删除，已有历史记录只能停用。
17. 生产和仓库通知能查看待处理、已确认历史和全部记录；管理员入口能查看全部分类历史。
18. 库存报警能按零件设置为“不报警”或“低库存报警”，报警只提示不自动执行业务动作。
19. 库存销毁 / 不再使用能通过盘点清零或报废 / 销毁流程留痕，不删除库存流水。
20. 统计页面能按只读口径展示数量。
