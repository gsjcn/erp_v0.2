## 3. 公司真实业务定位

公司名称：

```text
江阴市百胜制冷设备有限公司
```

公司主业：

```text
钣金制造加工。
```

业务定位必须这样理解：

```text
1. 江阴市百胜制冷设备有限公司的主营业务是钣金制造加工。
2. 系统必须围绕“钣金制造加工企业”的实际业务来设计。
3. 公司可能服务多个行业、多个客户类型、多个产品方向。
4. 汽车零配件只是公司业务中的一部分，不是公司的唯一业务，也不是公司的主营定位。
5. 公司目前已有涉及汽车零配件的业务，并且这部分业务正在按照 IATF 16949 体系要求运行。
6. 系统必须既能服务普通钣金制造加工业务，也能严格支撑 IATF 16949 要求下的汽车零配件相关业务。
```

```text

不要把系统写成只适用于汽车零配件的专用系统。
正确理解是：主业是钣金制造加工；IATF 16949 与汽车零配件是其中重要的质量体系场景和业务场景。
```

## 4. 项目定位

本项目是面向江阴市百胜制冷设备有限公司的：

```text
钣金制造加工企业 IATF 16949 双结构数字化管理系统
```

系统不是普通 ERP。  
系统不是普通 MES。  
系统不是普通 QMS。  
系统不是只做几个管理页面。  
系统不是某一行业或某一产品方向的专用系统。  
系统不是把功能堆在一起的后台模板。

系统核心定位是：

```text
一套钣金制造加工业务执行系统
+
一层 IATF 16949 条款映射系统
+
一套审核证据库
```

也就是说：

```text
业务人员按实际钣金制造加工流程工作。
体系人员按 IATF 16949 条款查看符合性。
审核人员按证据库检查执行记录和闭环结果。
管理层按驾驶舱查看体系运行、质量绩效、过程绩效、库存状态、交付状态和风险状态。
```

系统必须做到：

```text
普通钣金制造业务可以正常运行。
涉及 IATF 16949 的业务可以严格按照体系要求运行。
业务记录可以支撑 IATF 条款。
IATF 条款可以反查业务证据。
审核证据可以追溯到原始业务记录。
主要制造过程可以被记录、控制、追溯、分析和改进。
```

## 5. 严格遵守的核心原则

开发本系统必须严格遵守以下原则：

```text
1. 不做两套系统。
2. 不做普通 ERP 换皮。
3. 不做某一行业或某一产品方向的专用系统。
4. 不把公司主营业务误写成某一行业或某一产品方向。
5. 不把用户举例的库存、工艺、不良品误解为全部业务范围。
6. 不做零散页面堆积。
7. 不复制 IATF 16949 标准正文。
8. 不为了快速完成而牺牲架构、质量、权限、日志、审批和证据逻辑。
9. 不允许制造屎山代码。
```

允许保存的 IATF 信息是：

```text
条款编号
条款名称
内部解释
是否适用
责任部门
对应过程
对应模块
对应文件
对应记录
对应证据
符合状态
整改状态
```

不允许做的事情：

```text

把标准原文硬编码进系统
把体系条款写成无法维护的静态页面
把业务模块写成和条款没有关系的孤岛
把系统写死成只适用于汽车零配件
```

## 6. IATF 16949 双结构核心思想

系统必须采用“双结构”。

### 6.1 业务视角

业务视角用于员工日常操作，回答：

```text
今天要做什么？
客户要求是什么？
订单是否评审？
材料是否合格？
工单如何生产？
工序如何报工？
库存如何流转？
不良如何处理？
文件是否受控？
批次能否追溯？
问题是否关闭？
```

### 6.2 条款视角

条款视角用于体系管理和审核，回答：

```text
这个业务活动支撑 IATF 哪些条款？
某个条款有哪些过程、文件、记录和证据？
哪个部门负责？
证据是否完整？
是否存在不符合、风险或待整改项？
```

### 6.3 证据视角

证据视角用于内审、客户审核、第三方审核和审核准备，回答：

```text
某个条款的证据在哪里？
某个过程的记录是否完整？
某个问题是否闭环？
某个文件是否受控？
某个批次能否正向和反向追溯？
某个整改是否验证有效？
```

### 6.4 数据链路

系统必须支持正向链路：

```text
IATF 条款
→ 公司过程
→ 受控文件
→ 业务模块
→ 业务记录
→ 审核证据
→ 整改闭环
```

系统也必须支持反向链路：

```text
业务记录
→ 对应业务模块
→ 对应公司过程
→ 对应 IATF 条款
→ 对应审核证据
```

## 11. 技术架构

推荐技术栈：

```text
前端：Vue 3 + TypeScript + Vite + Element Plus + ECharts
后端：NestJS + TypeScript + Node.js
数据库：PostgreSQL
ORM：Prisma
部署：Docker Compose
权限：RBAC + 菜单权限 + 按钮权限 + 数据权限
日志：操作日志 + 数据变更日志 + 审批日志
国际化：Vue I18n 或等效 i18n 方案
多端适配：PC Web + Mobile Web + Tablet/PDA Web，必须兼容苹果 iOS 和安卓 Android，优先采用响应式 Web / PWA 架构
扫码能力：支持二维码 / 条码扫码溯源，可通过移动端摄像头或外接扫码设备录入
文件存储：支持本地文件存储或 MinIO，后续可扩展对象存储
```

说明：

```text
HTML、CSS、JavaScript 是前端底层基础。
Vue 3 用于组织复杂页面和组件。
TypeScript 用于提高大型系统严谨性。
Node.js 是运行环境。
NestJS 是基于 Node.js 的后端框架。
Element Plus 用于企业后台 UI。
ECharts 用于驾驶舱、质量趋势、生产看板、库存分析。
PostgreSQL 用于企业级关系数据。
Prisma 用于数据库模型、迁移和类型安全访问。
移动端优先使用响应式 Web / H5 / PWA 思路，必须兼容 iOS Safari、iOS 微信/企业微信内置浏览器、Android Chrome、Android 微信/企业微信内置浏览器。不建议一开始拆成 iOS App、Android App 和 Web 多套系统，以免增加维护成本。
```

不建议用纯 HTML + JavaScript 直接写本项目，因为本系统存在复杂菜单、权限、审批、状态流转、批次追溯、证据映射和多业务模块。纯原生写法后期极易失控。

## 12. 中英文国际化 i18n 要求

系统未来可能涉及中文 / 英文版本切换。  
因此从第一阶段开始必须预留国际化架构。

开发策略：

```text
一开始建立 i18n 架构。
第一版默认中文。
第一版不要求完成全部英文翻译。
英文资源文件可以先建立基础占位。
后续逐步补齐英文翻译。
不允许中文版全部完成后再大规模重构页面文字。
```

前端要求：

```text
不允许在页面中大量硬编码中文。
菜单、按钮、标题、表格列名、表单标签、状态文字、提示信息必须通过 i18n 管理。
默认语言为 zh-CN。
预留 en-US。
推荐目录：
frontend/src/locales/zh-CN.ts
frontend/src/locales/en-US.ts
frontend/src/locales/index.ts
```

后端要求：

```text
错误提示、枚举显示名、导出报表标题、通知模板预留多语言能力。
可通过 Accept-Language 判断语言。
数据库中的业务数据，如客户名称、供应商名称、产品名称、检验记录、8D 描述，不强制翻译，保持用户输入内容。
```

需要支持多语言的内容：

```text
系统菜单

导出文件标题
```

不强制翻译的内容：

```text
客户名称

物料名称
图纸名称
工单备注
检验记录
```

## 13. 三端互通与移动扫码溯源要求

系统未来需要支持三端互通的双语网页版。

三端指：

```text
PC 端：管理人员、质量人员、体系人员、计划人员使用
手机端：支持苹果 iOS 和安卓 Android，用于现场扫码、拍照留证、移动查询、移动审批
平板 / PDA / 车间终端：车间报工、检验、仓储、扫码出入库使用
```

开发原则：

```text
第一版从架构上预留三端互通能力。
优先采用响应式 Web 架构，不要一开始拆成多个独立系统。
同一套后端 API 服务 PC、手机、平板 / PDA。
同一套权限体系服务不同终端。
同一套数据模型服务业务记录、溯源记录和审核证据。
页面必须考虑 iOS / Android 移动端窄屏展示，不允许只适配大屏电脑。
```

手机端必须支持的典型场景：

```text
扫码查看批次溯源
扫码查看产品信息
扫码查看工单信息
扫码查看检验状态
扫码查看不良记录
扫码查看文件 / SOP / 检验规范
扫码执行原材料出入库
扫码执行半成品转序
扫码执行成品入库 / 出货
扫码进行工序报工
扫码进行检验记录录入
扫码上传现场照片
扫码查看审核证据
```

扫码入口定位：

```text
扫码不是独立业务模块，而是库存、生产、质检、追溯、文件和证据查看的现场作业入口。
PC 端不应把“移动扫码”作为普通一级主导航长期占位，优先放在顶部快捷按钮、业务页面内操作按钮或现场作业入口。
手机 / PDA / 车间终端可以把“扫码作业”作为第一屏入口。
扫码后的落点必须回到具体业务页面或记录详情，例如批次、工单、工序任务、检验记录、SOP 或追溯链路。
```

扫码类型：

```text
二维码
条形码
物料标签码
产品批次码
工单码
工序流转卡码
成品标签码
出货标签码
设备码
工装码
计量器具码
文件二维码
```

扫码溯源必须支持：

```text
从二维码 / 条码进入批次详情
从批次详情查看正向追溯
从批次详情查看反向追溯
从批次详情查看库存流向
从批次详情查看工序记录
从批次详情查看检验记录
从批次详情查看不良记录
从批次详情查看出货客户
从批次详情查看审核证据
```

评测和验收时，溯源能力可能是重点检查项。  
因此任何与批次、工单、工序、库存、检验、不良、出货有关的记录，都必须考虑如何进入溯源链路。

## 14. 苹果 iOS 与安卓 Android 移动端兼容要求

第一版原则：

```text

同一套前端代码适配 PC、手机、平板 / PDA。
同一套后端 API 服务所有终端。
不要一开始拆成 iOS 原生 App、Android 原生 App 和 Web 三套系统。
如果未来确实需要原生 App，再基于现有 API 和业务能力扩展。
```

必须兼容的移动端环境包括：

```text
iOS Safari
iOS 微信内置浏览器
iOS 企业微信内置浏览器
Android Chrome
Android 微信内置浏览器
Android 企业微信内置浏览器
常见安卓工业平板 / PDA 浏览器
```

移动端能力要求：

```text
扫码溯源
摄像头拍照
相册选择图片
附件上传
移动端表单录入
移动端审批
移动端查看文件
移动端查看批次追溯
移动端查看审核证据
```

兼容性注意事项：

```text
摄像头调用要兼容 iOS 和 Android。
文件选择要兼容 iOS 和 Android。
二维码 / 条码扫描要优先使用浏览器可支持方案，必要时支持外接扫码枪输入。
```

```text
iPhone 上可以拍照上传并关联业务记录。
```

系统必须支持拍照留证、附件上传和证据归档。

```text

包装出货照片

```

移动端要求：

```text
手机端可以直接调用摄像头拍照上传。
手机端可以从相册选择图片上传。
上传时必须能关联业务单据。
```

```text

压缩包：zip, rar, 7z
其他企业常用附件格式，可按安全策略扩展
```

```text

公司过程

不良记录
8D / CAPA
工单

供应商
客户投诉
```

附件管理必须记录：

```text
附件编号
原始文件名
文件类型
文件大小
MIME 类型
存储路径
上传人
上传时间
关联模块
关联记录 ID
是否作为审核证据
证据状态
备注
```

安全要求：

```text
必须限制允许上传的文件类型。
必须限制单个文件大小，具体大小做成系统参数。
必须避免可执行脚本类危险文件直接上传和访问。
文件下载必须经过权限校验。
重要文件必须保留下载 / 预览 / 删除日志。
删除附件优先软删除。
```

证据要求：

```text
现场照片和附件不是普通文件，而是审核证据的一部分。
任何检验、不良、返工、报废、审核、整改、客户投诉记录，都必须能上传照片或附件。
证据库必须能查询这些附件，并能反查原始业务记录。
```

```text

CSV
PDF
```

导出规则：

```text
用户只能导出自己权限范围内的数据。
导出必须受角色权限、数据权限、部门权限、工厂权限控制。
重要数据导出必须记录日志。
```

```text
导出格式
导出条件
数据范围
文件名
IP 地址
设备类型

```

禁止：

```text
无权限用户导出数据

前端绕过权限直接下载文件
接口不校验权限直接返回全量数据
一次性导出超大数据导致系统卡死

```

大数据导出要求：


### 17.2 权限设计总原则

系统权限必须符合 IATF 16949 的职责分工、过程控制、文件控制、审核证据和数据完整性要求。

权限不能只分为“管理员”和“普通用户”。  
必须按照岗位职责、部门职责、过程职责、数据范围、审批权限和审核权限进行设计。

权限至少包含以下维度：

```text
菜单权限
按钮权限
数据权限
字段权限
导出权限
导入权限
审批权限
删除权限
证据查看权限
附件下载权限
系统配置权限
```

关键原则：

```text
只有授权管理层或指定责任人可以修改企业级重要数据。
车间工人不能直接修改数据库数据。
车间工人只能通过受控业务页面录入生产、检验、扫码、拍照等现场记录。
所有关键数据修改必须有操作日志。
重要数据修改必须走审批或版本控制。
已审核、已批准、已发布、已关闭的数据不能随意修改。
如确需修改，必须通过变更流程、反审核流程或更正流程，并保留原因和证据。
一个用户可以同时拥有多个角色。
一个账户可以同时拥有多个权限。
用户最终权限 = 用户所有角色权限的并集。
后续如支持用户直接权限，用户最终权限 = 角色权限并集 + 用户直接权限。
一人身兼数职时，不应创建超级角色，应给同一用户分配多个岗位角色。
临时职责应优先建立临时角色或临时授权记录，后续应支持有效期、撤销和日志。
```

## 20. 后端开发规范

后端必须遵守以下规则：

1. 使用 NestJS 模块化结构。
2. 每个业务模块独立 module、controller、service、dto。
3. Controller 只负责接口入口，不堆业务逻辑。
4. Service 负责业务逻辑。
5. DTO 必须做参数校验。
6. Prisma 负责数据访问。
7. 重要状态必须使用枚举，不要散落字符串。
8. 所有关键表必须有审计字段。
9. 关键操作必须写 operation_logs。
10. 删除操作优先使用软删除。
11. 查询必须支持分页、筛选、排序。
12. 常用查询字段必须考虑索引。
13. 不允许 N+1 查询。
14. 不允许重复查询。
15. 不允许无意义循环。
16. 不允许把大数据量全部拉到内存再处理。
17. 错误处理必须清晰。
18. 返回结构必须统一。
19. 库存扣减必须考虑事务。
20. 批次追溯必须保证数据一致性。
21. 不良品处理、库存消耗、工序报工必须使用事务保护。
22. 不允许为了快速完成而绕过权限、日志、审批和证据逻辑。
23. 业务逻辑要清楚分层，不允许把所有逻辑写进一个巨大 service。
24. 公共逻辑必须合理抽象，但不允许过度设计。
25. 后端错误提示、枚举显示名、导出报表标题和通知模板要预留多语言能力。
26. 所有新增、修改、删除、导出、下载、审批接口必须做后端权限校验。
27. 数据导出必须经过权限检查和数据范围过滤。
28. 车间工人等现场角色不得通过任何接口直接修改企业级重要数据。
29. 重要数据修改必须支持审批、版本或变更记录。
30. 文件上传必须统一通过 storage / attachment 服务处理，不允许各模块各写一套上传逻辑。
31. 扫码解析必须统一封装，不允许在多个页面重复解析二维码 / 条码逻辑。
32. 后端必须校验角色分配、批量权限修改、账号锁定 / 解锁 / 停用等 RBAC 管理权限。
33. 后端批量权限修改必须使用事务，并写入 operation_logs。
34. 后端计算用户权限时必须支持一个用户多个角色，并按最终权限并集判断。

## 21. 前端开发规范

前端必须遵守以下规则：

1. 使用 Vue 3 Composition API。
2. 使用 TypeScript。
3. 页面、组件、API、类型定义分层清晰。
4. 不要把大量业务逻辑堆在一个页面文件里。
5. 通用表格、状态标签、证据关联区、附件区、审批流组件要复用。
6. 路由集中管理。
7. API 路径集中管理。
8. 枚举和状态集中管理。
9. 菜单数据集中管理。
10. 表格必须支持分页、筛选、加载状态和空状态。
11. 表单必须有校验。
12. 错误提示必须清晰。
13. 页面风格保持企业级、简洁、严谨。
14. 不允许为了视觉效果写复杂低效代码。
15. 不允许复制粘贴大量重复页面。
16. 不允许散落硬编码状态值和路径。
17. 大页面必须拆分为合理组件。
18. 组件职责必须单一，命名必须清晰。
19. 页面布局必须清楚，不允许堆积式页面。
20. 能复用的交互和展示组件必须复用。
21. 页面文字必须尽量通过 i18n 管理，不允许大量硬编码中文。
22. 前端按钮显示必须受权限控制，但不能把前端隐藏按钮当作安全措施。
23. 下载、导出、删除、审批等危险操作必须有明确提示和二次确认。
24. 页面必须考虑 PC、手机、平板 / PDA 等不同屏幕宽度。
25. 移动端扫码、拍照、上传相关功能必须封装为可复用组件或服务。
26. 菜单、页面、按钮、导出、下载、审批、批量权限修改等功能入口必须根据当前用户权限开放、隐藏或禁用。
27. 一个账户拥有多个角色时，前端必须按最终权限并集显示可用功能。
28. 前端权限显隐只是体验控制，安全必须以后端校验为准。
29. 扫码入口必须按终端和业务场景布置：PC 端以快捷入口和业务内动作优先，手机 / PDA 端以现场扫码作业入口优先。
